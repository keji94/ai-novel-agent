# Fix Loop 闭环机制

> 诊断→修复→验证→收敛 的自动化闭环，确保大纲从"有结构性问题"迭代到"可以开写"。

## 核心流程

```
Round 1: full_diagnostic → 输出诊断报告（P0/P1/P2/P3 分级）
         ↓
修复阶段: 按 P0→P1 优先级直接修改大纲文件
         ↓
Round 2: focused_check（只检查修复过的卷）→ 验证修复 + 回归检测
         ↓
      P0>0 或 P1>2？→ 再修一轮（Round 3）
         ↓
收敛: P0=0 且 P1≤1 → 输出"可以开写"
```

## 收敛标准

| 级别 | 收敛条件 |
|------|---------|
| 可以开写 | P0 = 0 且 P1 ≤ 1 |
| 需要用户决策 | P0 = 0 但 P1 ≥ 2，或有框架间矛盾 |
| 必须再修 | P0 > 0 |

## 最大轮数

**3 轮**。第 3 轮仍未收敛 → 输出"遗留问题清单"，交用户决定。

## 修复优先级

| 优先级 | 修复范围 | 说明 |
|--------|---------|------|
| P0 致命 | 必须修 | 结构性缺陷，不修则骨架不成立 |
| P1 严重 | 应该修 | 重要节拍缺失/错位，影响追读或角色深度 |
| P2 中等 | 建议修 | 节奏/伏笔/配角问题，可在写作阶段逐卷处理 |
| P3 轻微 | 不修 | 优化建议，留待写作时自然消化 |

Fix Loop 只修 **P0 + P1**，不动 P2/P3（改动面控制）。

## 修复操作规范

### 修改文件范围

| 问题位置 | 需修改的文件 |
|---------|------------|
| 单卷内部 | 该卷 volume 文件 |
| 跨卷影响 | 卷文件 + 总大纲（情绪节拍/卷结构表） |
| 角色设定影响 | 卷文件 + 角色设定.md / 主角设定.md |

### 修改原则

1. **最小改动**：只改问题涉及的段落，不重构整个卷
2. **同步更新**：改了卷大纲就必须同步总大纲的情绪节拍和爽点列表
3. **不新增线头**：修复不能引入新的伏笔或角色（除非是 P0 级修复必须）
4. **保持字数平衡**：修复不应让某卷字数偏离规划超过 10%

### 爽点/转折同步检查清单

修改任何卷的章节规划后，必须同步更新：
- [ ] 该卷的"核心爽点"列表
- [ ] 该卷的"关键转折"列表
- [ ] 总大纲的"每卷情绪节拍"
- [ ] 受影响的伏笔埋设/回收列表（如涉及）

## 回归检测

每轮修复后，必须检查：

| 检查项 | 方法 |
|--------|------|
| 新增设定与已有世界观是否矛盾 | 检查修复内容是否与前后卷的设定冲突 |
| 修复是否改变了其他卷的伏笔回收 | 检查总大纲的伏笔追踪表 |
| 情绪节拍是否与实际章节一致 | 比对总大纲情绪节拍与卷大纲实际内容 |
| 字数/章数是否仍然平衡 | 检查卷结构表的章数是否仍合理 |
| 新增角色是否与已有角色功能重叠 | 检查角色设定.md |

如果回归检测发现问题 → 升级为新的 P1，纳入本轮修复。

## 报告格式

### Round 1（全量诊断）

使用 SOUL.md 中定义的"结构诊断报告"格式，6 框架逐项检查。

### Round 2+（验证报告）

```
## Round {N} 验证报告

### 修复验证
| # | 原问题 | 修复内容 | 验证结果 |

### 回归检测
| 检查项 | 结果 | 说明 |

### Round {N} 问题扫描
- P0: {n} 个
- P1: {n} 个
- P2: {n} 个（残留）
- P3: {n} 个（残留）

### 收敛判定
{可以开写 / 需要用户决策 / 必须再修}
```

## 触发条件

- 用户说"审核大纲"/"诊断大纲" → 先跑 Round 1
- 用户说"修一下"/"fix loop" → 进入修复闭环
- 用户说"editorial_review" → 走编辑审读流程（不经过 fix loop）
