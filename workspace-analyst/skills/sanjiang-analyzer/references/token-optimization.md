# Token 优化策略

## 问题背景

章节内容较长，直接传入 AI 会导致 token 消耗过大。需要采用优化策略。

---

## 优化策略

| 策略 | 说明 | 效果 |
|------|------|------|
| **分段处理** | 将20章分成2组，每组10章单独分析后合并 | 避免超长上下文 |
| **缓存结果** | 已分析的结果缓存，避免重复调用AI | 避免重复计算 |
| **并行处理** | 多本书同时分析，提高效率，最高并发 2 | 提升速度 |
| **每本书一个会话** | 每本书都新开一个会话，避免历史对话导致 token超长 | 防止上下文膨胀 |
| **文档按需加载** | API 文档按需读取，不一次性加载全部 | 减少 ~60% |

## 使用方法

当需要执行某个操作时，使用 `read` 工具按需加载对应文档：

```python
# 示例：触发采集任务时
read("references/api-fetch-cache.md")

# 示例：进行 AI 分析时
read("references/ai-analysis.md")
```

---

## 注意事项

1. **不要一次性加载全部文档**：按需加载
2. **缓存已加载的文档**：同一会话内避免重复读取
3. **合理控制章节数量**：maxChapterCount 默认 20 章已足够
4. **使用新会话分析新书**：避免历史对话膨胀
